# AGENTS.md

Guidance for coding agents working in `@heyhuynhgiabuu/pi-search`.

## Repo purpose

This package ships five research tools for the Pi coding agent:

- `websearch` ‚Äî open-web search via Exa (direct REST or MCP fallback)
- `codesearch` ‚Äî code/library search via Exa (direct REST or MCP fallback)
- `context7` ‚Äî up-to-date library docs
- `deepwiki` ‚Äî ask questions about any public GitHub repo
- `web_fetch` ‚Äî extract full text from a URL

It is TypeScript-first and builds with `tsc` to `dist/`. The build is required because
`package.json#pi.extensions` references the compiled output.

## Project conventions

- ESM imports throughout.
- Local imports use the `.js` extension (NodeNext style).
- One responsibility per file; validation, provider calls, formatting, and registration stay separated.
- Preserve stable error codes (see `src/errors.ts`). Adding codes is non-breaking; renaming is.
- Prefer `src/exa/params.ts` constants (`DEFAULT_NUM_RESULTS`, `MAX_HIGHLIGHTS_MAX_CHARACTERS`, etc.) over hardcoded values in tool files.
- Formatting is handled by Biome (tabs, 120 cols).
- TypeScript is strict; no emit during typecheck.
- Tests are in `tests/`, one file per module under `tests/tools/` for per-tool tests.

## Required validation before finishing

Run the canonical checks:

```bash
make check
```

This runs `biome check .` (lint + format), `tsc --noEmit`, and `vitest --run --coverage` (with thresholds).

Equivalent raw commands:

```bash
npm run format
npm run lint
npm run typecheck
npm run test
```

## Release workflow

This repo uses Changesets. Typical release flow:

```bash

# Add a changeset describing the change
npx changeset
# Bump version + update CHANGELOG.md
make version-packages
git add -A && git commit -m "chore: release"
git push origin main
make release        # or: npm publish --access public
```

## Agent workflow recommendations

1. Read `README.md` and the relevant `src/tools/*.ts` before making behavior changes.
2. Read the matching test file before editing a module ‚Äî keep test coverage at the configured thresholds.
3. Make minimal, localized changes. Avoid editing `src/index.ts` for behavior; it should only wire tools.
4. If user-visible behavior changes, update `README.md` and add a changeset in the same change.
5. If you add a new tool, add a new file under `src/tools/`, a new test under `tests/tools/`, and update `TOOL_NAMES` in `src/index.ts`.
6. Do not commit `dist/` unless you intentionally want to ship a build (it's in `.gitignore`).
7. Do not add a build step beyond `tsc` unless explicitly requested.

## Common pitfalls

- Do not let raw error strings reach the model ‚Äî use `buildErrorResult(toPiSearchError(error))` in every `execute()`.
- Do not import MCP modules statically inside tool files ‚Äî use dynamic `import("../mcp/client.js")` to keep tools tree-shake-friendly.
- Do not hardcode the Exa API URL or search mode strings ‚Äî they live in `src/exa/client.ts` and `src/types.ts`.
- Do not let `onUpdate` callback be missing-handled ‚Äî every multi-step tool should stream progress.
- Domain include/exclude validation is strict; preserve that behavior.
- Tool output shape and `details` payload are tested; update tests when changing them.

## Good default checklist for most tasks

- inspect relevant source file(s)
- inspect matching test file(s)
- make the minimal change
- update docs if behavior changed
- add a changeset
- run `make check`
- summarize changed files and any follow-up steps
