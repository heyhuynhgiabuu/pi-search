# v0.2.4 — Architecture hardening + TUI rendering

This release is a pure refactor + new opt-in capabilities. The public API is **unchanged** — same five tools (`websearch`, `codesearch`, `context7`, `deepwiki`, `web_fetch`), same env/file resolution, same `disabledTools` config. **No action required to upgrade.** Bumping patch because no user-facing behavior changed.

## What's new under the hood

### Modular architecture
The previous ~960-line `src/index.ts` is gone. Source is now split into focused modules with a clear change map (see `AGENTS.md`):

```
src/
├── index.ts          # extension entrypoint (wiring only)
├── config.ts         # env + ~/.pi/pi-search.json resolution
├── errors.ts         # coded errors with stable code strings
├── types.ts          # shared types
├── mcp/client.ts     # JSON-RPC 2.0 MCP client
├── exa/client.ts     # direct REST client for api.exa.ai
├── exa/params.ts     # Exa parameter normalization
└── tools/            # one file per tool + shared citations + render
```

### TUI rendering (new — opt-in via host's expanded/collapsed toggle)
Every tool now ships a `renderResult` that produces a Markdown widget:

- **Collapsed view** — a ~1.5k-char preview is shown by default. User toggles Ctrl+O to expand.
- **Expanded view** — full content, paginated in 8k-char chunks with `[page N/M — X chars total]` footers.
- **Citations** — when expanded, a `**Sources**` block is appended with markdown links to the source URLs.
- **Theme-aware** — uses the host's `theme.fg("mdHeading", …)` colors via a small `toMarkdownTheme()` adapter that bridges pi-coding-agent's `Theme` to pi-tui's `MarkdownTheme`.
- **Per-tool polish**:
  - `websearch` / `codesearch` — numbered titles + URLs + highlight snippets + Sources footer
  - `context7` — extracts the `## <title>` and renders it as a heading
  - `deepwiki` — prepends a muted `DeepWiki · <repo>` tag
  - `web_fetch` — prepends a muted URL header

The TUI rendering is **adapted from PR #1 by x4cc3** (earendil fork). The fork isn't required — we use the mainline `@mariozechner/pi-coding-agent` + `pi-tui` APIs.

### Coded errors
Every tool now returns a stable error envelope that the model can branch on:

```ts
{ content: [...], details: { error: { code, message, details? } } }
```

Stable codes: `validation_error`, `config_error`, `provider_error`, `mcp_error`, `mcp_timeout`, `mcp_unavailable`, `exa_unauthorized`, `exa_rate_limited`, `fetch_error`, `fetch_timeout`, `aborted`, `internal_error`. Tools with narrower validation (`websearch`, `codesearch`) may also emit `missing_query`, `invalid_num_results`, `invalid_domain`, `domain_conflict`, `mixed_date_filters`, `invalid_date_range`.

### Streaming progress
Multi-step tools now emit `onUpdate` progress events. For example, `websearch` with 3 queries:

```
Exa searching 1/3: <query 1>
Exa searching 2/3: <query 2>
Exa searching 3/3: <query 3>
```

### Direct REST for Exa (opt-in)
Set `EXA_API_KEY` (or `PI_SEARCH_USE_REST=true`) to route `websearch` and `codesearch` through `https://api.exa.ai/search` directly. This unlocks the full Exa feature surface that the MCP server doesn't expose:

- `searchType`: `auto` | `neural` | `instant` | `deep` | `deep-reasoning` | `deep-max`
- `recencyFilter`: `day` | `week` | `month` | `year` (mutually exclusive with explicit date bounds)
- `startPublishedDate` / `endPublishedDate` — ISO date or datetime
- `includeDomains` / `excludeDomains` / `domainFilter` (mix with `-` prefix to exclude)
- `highlightsMaxCharacters` — 200..4000

When `EXA_API_KEY` is **not** set, the existing MCP path is used (zero-config behavior preserved).

## Upgrade notes

- The default config path `~/.pi/pi-search.json` is **unchanged** — no migration needed.
- New env vars: `PI_SEARCH_USE_REST`, `PI_SEARCH_CONFIG_PATH` (optional).
- New config keys: `useRestForExa` (boolean, default `false`; auto-enabled when `EXA_API_KEY` is set), `mcpTimeoutMs` (default 30000).
- The `disabledTools` config field is unchanged.
- The 5 tool names are unchanged.
- TUI rendering is automatic when the host supports `renderResult`; no opt-in needed.
